## System Overview

M.V.R.ESPRINT1 is a Rust-based deterministic assurance layer for energy-grid evaluation workflows, providing cryptographically verifiable event reconstruction, tamper-evident audit trails, and deterministic CSV verification.

**Core Capabilities**:
- Cryptographic attestation record generation
- Tamper-evident log chaining
- External verification of integrity
- Deterministic CSV normalization and chain verification
- Demo and dashboard support for pilot evaluation

**Target Environment**: Rust-based evaluation workflows on Linux or WSL. Ubuntu 24.04 under WSL2 with `rustc 1.94.1` and `cargo 1.94.1` is the documented verified environment.

**Market Operations Integration**:
- **Normal Operation**: deterministic feasibility and chain-verification workflows around SCED-style market data
- **Saturation Detection**: modeled in the constraint system and demo pipeline
- **L7 Transitions**: represented as explicit constrained outcomes rather than silent state mutation

**Regulatory Advantage**: unlike systems that hide infeasibility inside tolerance handling, M.V.R.ESPRINT1 is oriented toward explicit admissibility, reproducible outputs, and auditable evidence.

## Architecture

### High-Level Architecture

```text
[Input Artifact / Scenario]
    |
[Normalization / Deterministic Evaluation]
    |- sovereign_kernel
    |- constraint_system
    |- sced_offer_chain
    |- demo_pipeline
    |
[Attestation / Verification]
    |- verifier
    |- sovereign_trace
    |- guardian
    |
[Operator Review]
    |- demo CLI
    |- dashboard
```

### Module Structure

- **sovereign_kernel**: attestation generation, signer selection, and execution scaffolding
- **tlbss_integrity_engine**: grid physics and stability calculations
- **regulatory_policy**: NERC/CIP compliance mappings
- **audit_guardian**: boundary condition monitoring
- **constraint_system**: TLBSS-compliant constraint evaluation and admissibility checking
- **sovereign_trace**: immutable audit trail management
- **sovereign_bus**: unified communication channel
- **universal_frontend**: multi-language IR model with scaffold frontends
- **crypto_pipeline**: cryptographic binding pipeline with placeholder execution stages
- **sced_offer_chain**: deterministic CSV verification and chain reporting
- **demo_pipeline**: scenario modeling and demo output construction

### Distributed Deterministic Faction Split

The implementation is partitioned into three deterministic factions:

- **M.V.R.E (Central Control Center)**
  - Consumes canonical deterministic outputs for operational coordination
  - Does not alter canonical records generated by `sprint1`

- **sprint1 (Deterministic Binding Substrate)**
  - Performs schema lock enforcement, normalization, deterministic sorting, and hash-chain construction
  - Publishes identical canonical payloads to both M.V.R.E and Guardian

- **Guardian (Audit-Only Authority)**
  - Replays and verifies chain integrity independently from operational state
  - Cannot generate or mutate system state; it certifies admissibility and integrity only

Deterministic guarantee: for identical input CSV and locked schema, all factions converge on identical `final_chain_hash`.

### Execution Model

- **Deterministic**: identical inputs produce identical outputs
- **Memory Safe**: `#![deny(unsafe_code)]`
- **Feature-Gated TPM**: optional hardware-backed path behind the `tpm` Cargo feature
- **Local Verification First**: current documented workflows run as local CLI or dashboard processes

## Component Specifications

### Sovereign Kernel

- **Language**: Rust 2021
- **Role**: deterministic attestation and execution scaffold
- **Signer Selection**: runtime-selected via `SIGNER_MODE`
- **Attestation Type**: `AttestationRecord`

### Attestation Pipeline

- **Hash Algorithm**: SHA-256
- **Signature Mode**: simulated fixed-key signer by default, optional TPM-backed path
- **Chain Integrity**: hash-linked records with `prev_hash`
- **Timestamp Precision**: Unix epoch seconds

### Verifier

- **Validation Checks**:
  - JSON decoding of `Vec<AttestationRecord>`
  - signature verification
  - hash-chain integrity
  - timestamp monotonicity

- **Input**: JSON file path passed on the command line
- **Output**: pass/fail process result with human-readable status text

### SCED Offer Chain Verifier

- **Input**: CSV file path
- **Command Form**: `cargo run --bin sced_chain -- verify <input.csv> [expected_hash]`
- **Validation Checks**:
  - CSV schema match
  - required values present
  - boolean and numeric parsing
  - duplicate primary-key detection
  - chain continuity and expected-hash validation

- **Output Forms**:
  - informational status lines
  - JSON `VerifierReport`
  - terminal `[PASS]` or `[FAIL]` summary

### Constraint System

- **Purpose**: TLBSS-compliant constraint evaluation and admissibility certification
- **Architecture**: pure-function style evaluation and binary admissibility decisions
- **Constraints**: ramp rates, capacity limits, regulation headroom and footroom, time-coupled trajectories
- **Principle**: Guardian certifies transitions and does not mutate state
- **Status**: implemented as a core modeling component; performance claims are design intent rather than benchmark-certified in this document

### Signer Modes

- **Simulation**: deterministic software signing for testing
- **TPM**: optional feature-gated path with scaffolded TPM internals

### Universal Execution Layer Status

- **Rust path**: native IR, codegen, and kernel execution path is present
- **Python/JavaScript/C#/Go frontends**: scaffold frontends currently return not-implemented parse errors
- **crypto_pipeline**: contains placeholder normalization, execution, and translation stages
- **Operational implication**: treat the universal execution layer as an in-repo scaffold, not a completed production ingestion surface

### Dashboard

- **Framework**: Axum 0.7 on Tokio
- **Bind Address**: `127.0.0.1:3000`
- **Routes**:
  - `/`
  - `/health`
  - `/demo/:scenario`

## Security Specifications

### Cryptographic Primitives

- **Hash Function**: SHA-256
- **Signature**: simulated fixed-key signer by default; optional TPM feature path
- **Key Management**: software simulation or TPM 2.0 path
- **Chain Integrity**: hash-linked immutable records

### Threat Model

- **Threats Mitigated**:
  - log tampering
  - replay or continuity break within chained records
  - malformed CSV or inconsistent normalization inputs
  - unauthorized silent state mutation in audit-only flows

### Compliance

- **CIP-007**: system integrity protection
- **CIP-010**: configuration integrity
- **CIP-013**: supply-chain and trust-boundary framing
- **Control-to-Evidence Mapping**: `docs/compliance_mapping.md`
- **Access Control Boundaries**: `docs/compliance_mapping.md` (Section 2)

### Audit Capabilities

- Immutable-style record chaining
- External verification through CLI tooling
- Deterministic reproduction of verification outcomes
